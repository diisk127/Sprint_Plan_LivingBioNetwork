## 2. วิเคราะห์งานที่ต้องทำ

### 2.1 งานแต่ละชิ้นยากแค่ไหน

| ชิ้นงาน | ความยาก | ความเสี่ยง | เวลาที่ใช้ |
|:---|:---:|:---|:---|
| สร้าง BioNode | ปานกลาง | ต่ำ | 8–10 ชม. |
| สร้าง BioNetworkState | ปานกลาง | ต่ำ | 6–8 ชม. |
| ระบบหาเส้นทาง (Routing) | ยากมาก | สูง — ต้องทำก่อนส่วนอื่น | 20–25 ชม. |
| ระบบส่งสัญญาณ | ยาก | ปานกลาง | 12–15 ชม. |
| ระบบเข้ารหัส | ยาก | ปานกลาง | 10–12 ชม. |
| ระบบบันทึก Log | ง่าย | ต่ำ | 4–6 ชม. |
| เขียน Test | ปานกลาง | ปานกลาง | 8–10 ชม. |
| เขียนเอกสาร | ง่าย | ต่ำ | ทำต่อเนื่อง |

**รวมเวลาทั้งหมด:** ประมาณ 68–86 ชั่วโมง
**เวลาที่ทีมมี:** 150 ชั่วโมง (มีเวลาสำรองเหลือพอ)

### 2.2 ต้องทำอะไรก่อน-หลัง

งานที่ต้องทำตามลำดับ (ทำข้ามไม่ได้):
```
BioNode → ระบบเครือข่าย → ระบบหาเส้นทาง → ระบบส่งสัญญาณ → ระบบเข้ารหัส
```

งานที่ทำพร้อมกันได้:
- เขียน Log ทำควบคู่กับ BioNode ได้เลย
- เขียน Test ทำควบคู่กับทุกชิ้นงาน
- เขียนเอกสารทำต่อเนื่องตลอด

### 2.3 ความเสี่ยงที่ต้องระวัง

| ความเสี่ยง | ผลกระทบ | วิธีรับมือ |
|:---|:---|:---|
| สัญญาณวนลูปไม่รู้จบ | สูง | เพิ่มระบบตรวจจับการวนซ้ำ |
| คำนวณ Routing ผิดพลาด | ปานกลาง | เขียน Test ตรวจสอบทุกกรณี |
| ระบบช้าเมื่อมี Node เยอะ | ปานกลาง | จำกัดขนาดไว้ก่อนใน Sprint Alpha |
| รวมระบบแล้วพัง | สูง | ทดสอบทีละส่วนก่อนรวม |

## 3. ทบทวนโครงสร้างระบบ (Architectural Review)

### 3.1 ผลการตรวจแต่ละส่วน

| ส่วนของระบบ | ผล | หมายเหตุ |
|:---|:---:|:---|
| Infrastructure Layer |  ผ่าน | BioNode ออกแบบชัดเจน พร้อมพัฒนา |
| Evolution Engine |  ผ่านมีเงื่อนไข | ต้องกำหนดค่า Threshold ให้ชัดก่อน |
| Existence Communication Layer |  ผ่าน | โครงสร้างข้อมูลครบถ้วน |
| Structural Privacy Layer |  ผ่าน | ระบบ Genetic Key ออกแบบดี |
| Human Awareness Layer |  ผ่านมีเงื่อนไข | ต้องออกแบบ Interface ให้ละเอียดขึ้น |

### 3.2 สิ่งที่ต้องแก้ก่อนเริ่ม Sprint Beta

1. กำหนดค่า Threshold สำหรับตัดสินว่าเส้นทางไหนแย่เกินไป
2. กำหนดค่าระดับ Traffic ที่ถือว่าหนาแน่นเกินไป
3. ออกแบบหน้าจอเชื่อมต่อกับผู้ใช้ให้ชัดเจนขึ้น

### 3.3 การตัดสินใจสำคัญ

| การตัดสินใจ | เหตุผล | สถานะ |
|:---|:---|:---:|
| ใช้ Python ในการพัฒนา | ทีมคุ้นเคย พัฒนาเร็ว | ✅ |
| ใช้ Dictionary เก็บ Node | ค้นหาได้เร็วกว่า List | ✅ |
| ใช้ Greedy Algorithm ก่อนใน Sprint Alpha | ง่ายกว่า ค่อยพัฒนาเป็น Dijkstra ใน Beta | ✅ |
| ใช้ String Matching สำหรับ Key ก่อน | ทำง่าย ค่อยซับซ้อนขึ้นใน Sprint Delta | ✅ |

## 4. แผนงานแต่ละ Sprint

### Sprint Alpha — สร้าง BioNode พื้นฐาน

**เป้าหมาย:** BioNode เชื่อมต่อและส่งสัญญาณหากันได้

| งาน | คนรับผิดชอบ | ต้องเสร็จวันที่ | ถือว่าเสร็จเมื่อ |
|:---|:---|:---:|:---|
| เขียน BioNode | ณัฏฐชัย | Day 3 | ส่ง-รับสัญญาณได้ ไม่วนลูป |
| เขียน BioNetworkState | ณัฏฐชัย | Day 4 | เพิ่ม Node และ Fiber ได้ |
| เขียนระบบหาเส้นทาง | ณัฏฐชัย + ภควัฒน์ | Day 7 | เลือกเส้นทางดีที่สุดได้ |
| เขียนระบบ Log | ณัฏฐชัย | Day 5 | บันทึกสถานะได้ ไม่กระทบระบบ |
| เขียน Test | กัญญาวีร์ | Day 8 | ผ่านมากกว่า 60% |

---

### Sprint Beta — ระบบเติบโตอัตโนมัติ

**เป้าหมาย:** เครือข่ายปรับโครงสร้างตัวเองได้เมื่อสถานการณ์เปลี่ยน

| งาน | คนรับผิดชอบ | ถือว่าเสร็จเมื่อ |
|:---|:---|:---|
| ระบบลดความหนาแน่นเมื่อไม่ใช้งาน | ณัฏฐชัย | ค่า Density ลดลงถูกต้อง |
| ระบบเพิ่มความหนาแน่นเมื่อใช้งานหนาแน่น | ณัฏฐชัย | ค่า Density เพิ่มขึ้นถูกต้อง |
| ระบบสร้างเส้นทางใหม่อัตโนมัติ | ณัฏฐชัย + ภควัฒน์ | สร้างเส้นทางใหม่ได้จริง |
| วัดประสิทธิภาพก่อน-หลัง | กัญญาวีร์ | เห็นความแตกต่างชัดเจน |

---

### Sprint Gamma — ระบบส่งสัญญาณชีวภาพ

**เป้าหมาย:** ส่งข้อมูลแบบมีโปรโตคอลและตรวจสอบความถูกต้องได้

| งาน | คนรับผิดชอบ | ถือว่าเสร็จเมื่อ |
|:---|:---|:---|
| ออกแบบโครงสร้าง BioPacket | ภควัฒน์ | กำหนดทุก Field ครบ |
| เขียนระบบเข้ารหัส-ถอดรหัสข้อมูล | ณัฏฐชัย | เข้ารหัสแล้วถอดกลับได้ถูกต้อง |
| เขียนระบบส่ง Packet | ณัฏฐชัย | ส่งถึงปลายทางได้ |
| เขียนระบบตรวจสอบว่าข้อมูลครบถ้วน | กัญญาวีร์ | จับ Packet เสียหายได้ |

---

### Sprint Delta — ระบบความเป็นส่วนตัว

**เป้าหมาย:** เฉพาะ Node ที่มี Key ถูกต้องเท่านั้นที่อ่านข้อมูลได้

| งาน | คนรับผิดชอบ | ถือว่าเสร็จเมื่อ |
|:---|:---|:---|
| ออกแบบรูปแบบ Genetic Key | ภควัฒน์ | กำหนด Format ชัดเจน |
| เขียนระบบถอดรหัสเฉพาะผู้มีสิทธิ์ | ณัฏฐชัย | Node ที่ Key ไม่ตรงเข้าไม่ได้ |
| เขียนระบบส่งข้อมูลเฉพาะผู้รับที่กำหนด | ณัฏฐชัย | ส่งได้เฉพาะ Node ที่มีสิทธิ์ |
| ทดสอบการโจมตีระบบ | กัญญาวีร์ | ทดสอบผ่านทุก Scenario |

## 5. หน้าที่ของแต่ละคน

**กวินธิดา (Product Owner)** — กำหนดว่าอะไรสำคัญกว่า อนุมัติงาน ตรวจสอบว่าได้ตามที่ต้องการ

**ภควัฒน์ (Architect)** — ออกแบบโครงสร้างก่อนเสมอ ตรวจโค้ดของณัฏฐชัยว่าตรงกับที่ออกแบบไว้

**กัญญาวีร์ (QA)** — เขียนแผนทดสอบ ทดสอบทุกชิ้นงาน รายงาน Bug ให้ณัฏฐชัยแก้

**ณัฏฐชัย (Engineer)** — เขียนโค้ดทุกอย่างตามที่ภควัฒน์ออกแบบ แก้ Bug ตามที่กัญญาวีร์รายงาน

**ธนนันต์ (Scrum Master)** — ดูแลการประชุมทุกวัน จัดการปัญหาที่ทีมติดขัด อัปเดตเอกสาร

## 6. ถ้าเกิดปัญหา

**ทำงานช้ากว่ากำหนด** → ตัดฟีเจอร์ที่ไม่จำเป็นออก เพิ่ม Pair Programming
